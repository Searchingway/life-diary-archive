足迹模块设计方案（v0.2 准备稿）

一、目标
足迹模块的定位不是地图软件，而是“按日期记录去过哪里、看到了什么、当时的感受”，并且和当天日记形成关联。

这一版我准备优先完成：
1. 新建足迹
2. 查看历史足迹
3. 编辑和保存足迹
4. 删除足迹
5. 按日期 / 地点名 / 感悟搜索
6. 插入图片
7. 足迹与当天日记双向跳转
8. 足迹统计图


二、我准备怎么做
我会尽量复用现在已经稳定的日记模块做法，不另起一套复杂架构，核心思路是：

1. 数据结构沿用“每条记录一个目录”
日记现在已经是：
- 一个 id 对应一个目录
- 目录里存 json / markdown / images

足迹我会保持同样思路，避免后面维护两套完全不同的存储方式。

2. 关系采用“按日期关联为主，id 关联为辅”
足迹天然必须绑定日期，所以最稳妥的是先用 date 做主关联。
如果以后需要更精确跳转，也可以额外保存 related_entry_id。

3. UI 先做成和日记相似的工作流
也就是：
- 左侧列表
- 中间编辑区
- 顶部操作按钮
- 下方图片区

这样用户不用重新学一套操作逻辑，代码也能直接复用现在的保存、图片、删除、未保存提示这些模式。

4. 不把所有逻辑硬塞进 main_window.py
现在日记功能已经比较完整了，如果足迹继续全部堆在一个窗口文件里，后面会越来越难维护。
所以我准备在保持现有功能不动的前提下，给足迹拆独立的数据层和界面层。


三、建议的数据结构
建议新增一个足迹模型，字段如下：

id
- 足迹唯一 id

date
- 足迹日期
- 这是和日记关联的核心字段

place_name
- 地点名称
- 例如“青秀山”“公司楼下咖啡店”

thought
- 这一段足迹感悟
- 相当于足迹的正文

images
- 图片列表
- 结构和日记图片保持一致
- 每张图仍然支持备注名

related_entry_id
- 可选
- 如果当天已经有明确关联的日记，可以保存对应日记 id
- 没有时允许为空

created_at
- 创建时间

updated_at
- 更新时间


四、建议的本地目录结构
我准备在当前数据目录下增加：

data/Diary/footprints/
  <footprint_id>/
    footprint.json
    thought.md
    images/
      xxx.jpg

其中：
1. footprint.json 保存元数据
2. thought.md 保存足迹正文
3. images/ 保存当前足迹的图片

这样好处是：
1. 和日记目录结构一致
2. 后续做备份或交换包很顺
3. 手工查看文件也容易


五、界面准备怎么做
我准备把主界面升级成“两大页签”或“两大模块切换”：

方案优先级：
1. 优先方案：QTabWidget
2. 备选方案：左侧加模块切换按钮

我更倾向于 QTabWidget，原因是改动更稳：
1. 可以保留当前日记页不大动
2. 足迹页单独做
3. 后面加读书笔记也容易继续扩

足迹页会包含这些区域：

左侧：
- 足迹列表
- 搜索框
- 新建足迹按钮
- 删除当前足迹按钮

中间：
- 日期
- 地点名称输入框
- 感悟编辑区
- 保存按钮
- 恢复已保存内容按钮
- 插入图片按钮

下方或右侧：
- 图片列表
- 图片备注输入框
- 图片预览
- 打开图片 / 移除图片

补充一个关联区：
- 打开当日日记
- 查看当天足迹数量
- 如果当天没有日记，可以提示“当天还没有日记”


六、和日记怎么双向关联
这是足迹模块里最关键的一块，我准备这样做：

1. 从足迹跳转到日记
- 根据足迹 date 找到同日期的日记
- 如果同一天只有一篇日记，直接打开
- 如果以后支持同一天多篇日记，再弹选择框

2. 从日记跳转到足迹
- 在日记页增加“查看当天足迹”入口
- 根据当前日记 date 查询所有同日足迹
- 可以直接切到足迹页，并定位到当天第一条足迹

3. 先避免做“双份冗余写入”
也就是说，不准备在日记 json 里硬写一个足迹 id 列表，同时又在足迹里再写日记 id 列表。
第一阶段先通过日期查询来建立双向关系，这样更稳，也不容易出现一边改了另一边没改的问题。

4. 如果后面确实需要强绑定
再增加 related_entry_id 字段，作为增强，而不是一开始就让两边都保存完整引用。


七、代码层我准备怎么拆
为了尽量少影响已经能用的日记模块，我准备按下面方式加：

1. models.py
新增：
- FootprintImage
- FootprintImageDraft
- FootprintEntry

如果后面发现 DiaryImage 和 FootprintImage 完全一样，也可以再做抽取，但第一步先保证可读性。

2. 新增 footprint_storage.py
职责：
- create_empty_footprint
- save_footprint
- load_footprint
- list_footprints
- list_footprints_by_date
- delete_footprint
- resolve_image_path

这样可以避免把 DiaryStorage 越堆越大。

3. 新增 footprint_widget.py 或 footprint_page.py
职责：
- 足迹列表和编辑区
- 图片列表与图片备注
- 发出“请求打开某天日记”的信号

4. main_window.py
职责：
- 只负责把“日记页”和“足迹页”装起来
- 处理两个页面之间的跳转

5. tests/
新增：
- test_footprint_storage.py
- 必要时补 GUI 的基础烟雾测试


八、第一批功能落地顺序
我准备按下面顺序做，这样最稳：

第 1 步：数据层
- 定义 FootprintEntry
- 写 FootprintStorage
- 写足迹的保存、读取、搜索、删除测试

第 2 步：足迹独立界面
- 把足迹页做出来
- 实现新建、编辑、删除、搜索、图片插入

第 3 步：双向关联
- 从足迹打开当日日记
- 从日记查看当天足迹

第 4 步：文档同步
- 更新 README
- 更新 yaoqiu.txt

第 5 步：补验证
- 跑 unittest
- 做一次手动界面流程验证


九、我会特别注意的几个点
1. 不破坏现有日记功能
足迹模块应当是新增，不应该把现在已经可用的日记功能搞乱。

2. 尽量复用图片处理逻辑
现在日记已经有图片插入、预览、备注、移除，这部分可以抽共性，但不会一上来就过度重构。

3. 关联逻辑先求稳，不求花哨
先把“按日期互相找到”做通，比一开始追求复杂关系图更实用。

4. 继续坚持本地优先
所有足迹数据都直接落本地目录，不引入数据库依赖。


十、我建议的验收标准
如果做到下面这些，就可以认为足迹模块第一版可用了：

1. 能新建一条足迹
2. 能保存到本地并重新打开
3. 能按日期和地点搜索
4. 能插入图片并保存图片备注
5. 能删除足迹
6. 能从足迹跳到当天日记
7. 能从日记查看当天足迹


十一、结论
我的实现策略是：
先做一个“和日记模块风格一致、结构一致、能真正使用”的足迹 MVP，
重点先把“CRUD + 图片 + 日期绑定 + 与日记双向关联”做扎实，
暂时不碰地图、统计和复杂可视化。

这样做的好处是：
1. 风险低
2. 开发快
3. 和你现在这套项目最匹配
4. 后面继续扩到读书笔记、轻计划也更顺
